# SQL Performance Optimization — Oracle & PostgreSQL

**Domain:** Database Performance Engineering · Query Optimization · Index Strategy  
**Stack:** Oracle 12c Release 2 · PostgreSQL 13 (Aurora)  
**Environment:** Financial services — regulatory reporting and core transaction processing  
**Improvement:** Regulatory report: 45 min → 2.8 min · Nightly batch: 8 hours → 1.2 hours

---

## A. Business Context

### The Enterprise Problem

A financial services firm operating a loan portfolio management system was generating critical quarterly regulatory reports — required for submission to the national financial regulator — via a Java Spring Batch process backed by Oracle 12c R2. The system also ran a nightly batch job that recalculated risk scores, accrual interest, and provisioning amounts for approximately 2.4 million loan accounts.

Both processes were experiencing escalating runtime that had grown year-over-year as the portfolio expanded. At the time of engagement:

- The quarterly regulatory report was taking **45 to 67 minutes** to generate
- The nightly batch was running **8 to 10 hours** — its window was 11 PM to 7 AM
- The batch was routinely running past its window into the morning, causing failures in downstream processes that consumed its output at 7 AM
- The regulatory report ran against the same Oracle instance as the OLTP transaction processing — long-running report queries were acquiring row-level locks and blocking loan origination transactions during business hours

### Why Optimization Was Necessary

The business impact was direct:

- **Regulatory compliance risk:** Reports submitted to the regulator must be generated by end of business on the quarterly deadline. Reports consistently running over 45 minutes created operational risk — any additional slowdown during deadline week would result in late submission
- **Operational batch failure:** When the nightly batch overran its window, the 7 AM downstream system (a third-party portfolio reporting vendor) received no updated data, requiring manual data correction procedures that consumed 3–4 hours of analyst time
- **Transaction processing degradation:** Report queries on the shared Oracle instance caused measurable latency spikes (P99 loan processing latency increased by 40% during report runs)

### Business Constraints

| Constraint | Detail |
|-----------|--------|
| **No hardware upgrade path** | Oracle instance was already on the largest available server tier; CPU was not the bottleneck |
| **No schema restructuring** | Third-party system had direct read access to the Oracle schema; column renames or table restructuring required third-party coordination (out of scope) |
| **Minimal application code changes** | Spring Batch jobs were maintained by a separate team with a separate release pipeline; changes required coordination and regression testing |
| **Reporting isolation preferred** | Business preferred reporting queries to not impact OLTP — a structural concern beyond pure query optimization |

### Diagnostic Methodology

Before any optimization was applied, a complete diagnostic pass was performed using:

1. **AWR (Automatic Workload Repository) reports** — 7-day snapshot period, 1-hour intervals, identifying top SQL by elapsed time, CPU time, buffer gets, and disk reads
2. **ASH (Active Session History)** — identifying wait event profiles for the report and batch SQL sessions
3. **DBMS_XPLAN.DISPLAY_CURSOR** — execution plans with actual row counts and predicate information for top SQL statements
4. **V$SQL and V$SQL_PLAN** — identifying plan instability (same SQL with multiple execution plans)
5. **DBA_HIST_SQL_STAT** — trend analysis on key SQL statements across the 7-day diagnostic window

This diagnostic phase took approximately 2 weeks before any changes were made to indexes or queries.

### Top SQL — Regulatory Report (pre-optimization)

The regulatory report was generated by a single Spring Batch job that ran 7 primary SQL queries. The top 3 by elapsed time:

**Query R-01 (Loan Portfolio Exposure Summary):**
```sql
-- Original query — 18 min average elapsed time
SELECT
    l.BRANCH_CODE,
    l.PRODUCT_TYPE,
    l.RISK_GRADE,
    SUM(l.OUTSTANDING_BALANCE) AS TOTAL_EXPOSURE,
    COUNT(l.LOAN_ID) AS LOAN_COUNT,
    SUM(CASE WHEN l.DAYS_PAST_DUE > 90 THEN l.OUTSTANDING_BALANCE ELSE 0 END) AS NPL_BALANCE,
    AVG(l.INTEREST_RATE) AS AVG_RATE
FROM LOAN_MASTER l
JOIN CUSTOMER_PROFILE c ON l.CUSTOMER_ID = c.CUSTOMER_ID
JOIN COLLATERAL_REGISTER cr ON l.LOAN_ID = cr.LOAN_ID AND cr.COLLATERAL_STATUS = 'ACTIVE'
JOIN BRANCH_REFERENCE b ON l.BRANCH_CODE = b.BRANCH_CODE
WHERE l.LOAN_STATUS IN ('ACTIVE', 'DELINQUENT', 'RESTRUCTURED')
  AND l.REPORT_DATE = :reportDate
GROUP BY l.BRANCH_CODE, l.PRODUCT_TYPE, l.RISK_GRADE
ORDER BY l.BRANCH_CODE, l.PRODUCT_TYPE;
```

**Execution plan problem identified:**
- `LOAN_MASTER` (2.1M rows) was doing a **full table scan** — the `REPORT_DATE` filter had an index (`IDX_LM_REPORTDATE`) but the optimizer was not using it
- Root cause: the `LOAN_STATUS IN (...)` predicate was using a **non-selective index** (`IDX_LM_STATUS`) — only 3 distinct values across 2.1M rows; cardinality estimate was wildly wrong due to stale statistics
- The `COLLATERAL_REGISTER` join was doing a hash join against a full scan — `cr.COLLATERAL_STATUS = 'ACTIVE'` filtered out 95% of rows, but no index existed on `(LOAN_ID, COLLATERAL_STATUS)`

**Query R-02 (Provisioning Calculation):**

A 340-line query joining 11 tables, with an inline view that duplicated the `LOAN_MASTER` full scan. The inline view result was not materialized — Oracle re-executed it for each outer query row.

**Query R-03 (Interest Accrual Aggregation):**

A cursor-based loop in a stored procedure processing 2.4M loan rows individually. Row-by-row processing — the classic "slow-by-slow" Oracle anti-pattern.

### Nightly Batch Analysis

The nightly batch was a Spring Batch `JpaPagingItemReader` reading `LOAN_MASTER` in pages of 1,000 rows, processing each loan, and writing to `ACCRUAL_LEDGER` and `RISK_SCORE_HISTORY`.

**Primary bottleneck identified:** The `JpaPagingItemReader` used an `ORDER BY LOAN_ID` clause to paginate. With 2.4M rows × 1,000-row pages = 2,400 Oracle SELECT queries, each using `ROWNUM` pagination. As the page offset increased, Oracle performed progressively larger index range scans plus a sort — the last page was taking 8 seconds while the first page took 0.2 seconds.

**Secondary bottleneck:** Individual `INSERT INTO RISK_SCORE_HISTORY` statements — approximately 2.4M individual INSERTs per night, not batched.

## C. Solution Evaluation

### Option 1 — Query Hints and Statistics Refresh

Refresh optimizer statistics (`DBMS_STATS`) and add Oracle query hints to force the correct execution plan on the 7 regulatory report queries.

**Pros:**
- Zero schema changes, zero application changes
- Immediate effect — statistics refresh can be run in hours
- Low risk, easily reversible

**Cons:**
- Query hints are a workaround for optimizer failures, not a fix — they become maintenance liabilities when data distribution changes
- Statistics refresh may not be sufficient for plan stability if cardinality estimates for `LOAN_STATUS` remain skewed (column has only 3 distinct values, histogram needed)
- Does not address the R-03 stored procedure cursor loop (requires PL/SQL changes)
- Does not address nightly batch pagination pattern (requires application changes)
- Does not address OLTP/reporting contention (structural problem)

**Risk:** Low  
**Cost:** 1 week of DBA time  
**Performance Impact:** Partial — likely 40–60% improvement on R-01/R-02; R-03 and batch unchanged

---

### Option 2 — Index Optimization + Query Restructuring

Add targeted composite indexes, rewrite the cursor-based R-03 stored procedure as a set-based operation, and resolve the nightly batch pagination pattern.

**Pros:**
- Addresses root causes, not symptoms — plans will remain stable as data grows
- No hardware changes, no schema restructuring (new indexes only, no column changes)
- Application changes limited to batch pagination strategy
- Composite index on `COLLATERAL_REGISTER(LOAN_ID, COLLATERAL_STATUS)` directly eliminates the most expensive hash join
- Partial indexes (Oracle function-based) reduce index maintenance overhead

**Cons:**
- Index additions require careful impact assessment — each new index increases DML overhead (INSERT/UPDATE to `LOAN_MASTER` and `COLLATERAL_REGISTER` will become slower)
- Query rewriting requires DBA + application team coordination
- Regression testing required on all affected jobs

**Risk:** Medium (DML performance impact must be measured)  
**Cost:** 3–4 weeks of DBA + developer time  
**Performance Impact:** High — targeted at root causes

---

### Option 3 — Materialized Views + Reporting Read Replica + Batch Redesign

Create pre-aggregated materialized views for the regulatory report, introduce an Oracle Active Data Guard read-only standby for report execution, and redesign the nightly batch to use a direct-path INSERT strategy.

**Pros:**
- Materialized views eliminate runtime aggregation — report reads pre-computed data
- Active Data Guard standby completely removes report/OLTP contention
- Direct-path INSERT for batch reduces DML overhead dramatically
- Most comprehensive performance improvement

**Cons:**
- Oracle Active Data Guard requires Enterprise Edition license — significant cost ($140K additional licensing)
- Materialized views require refresh strategy design (complete vs. fast refresh, synchronous vs. deferred)
- If materialized view refresh fails, regulatory report reads stale data — data integrity risk
- Direct-path INSERT bypasses redo logging options — must be carefully designed for recovery scenarios
- 8–12 weeks of work

**Risk:** Medium-High (licensing cost, refresh strategy complexity)  
**Cost:** $140K licensing + 8–12 weeks DBA/developer time  
**Performance Impact:** Maximum

---

## D. Final Architecture Decision

**Decision: Option 2 + selective elements of Option 3 (materialized views without ADG standby)**

### Why This Was Chosen

Option 2 alone is the right first pass. Fixing root causes (missing indexes, cursor loops, batch pagination) will produce the majority of improvement without licensing cost.

The reporting isolation problem (Option 3's ADG standby) is addressed through a different mechanism: the system is on an 18-month roadmap to Aurora PostgreSQL (as part of the AWS migration described in [Case Study 1](./legacy-java-aws-migration)). Rather than investing $140K in Oracle ADG, the reporting isolation is handled by:
1. Scheduling the quarterly report run outside OLTP business hours (2 AM, coordinated with the regulator's acceptance window)
2. Adding materialized views for the two most expensive aggregations (R-01, R-02) — addressing the aggregation cost without the ADG licensing

Option 3's direct-path INSERT strategy for the nightly batch is adopted as it fits within Option 2's scope.

### Principles Applied

- **Fix root causes, not symptoms** — statistics refresh + hints is a band-aid; composite indexes solve the plan stability problem permanently
- **Measure DML impact before deployment** — new indexes must be benchmarked on INSERT/UPDATE workloads in a staging environment with representative data volume
- **Incremental deployment** — changes applied in isolation with performance comparison before proceeding to next change
- **Cost-justified complexity** — ADG standby at $140K is not justified when scheduling change + materialized views achieves the same operational objective

## F. Implementation Strategy

### Phase 1 — Statistics and Histograms (Week 1)

No schema changes. Gather fresh statistics with extended histograms on the identified skewed columns:

```sql
-- Gather statistics with 100% sample and histograms
EXEC DBMS_STATS.GATHER_TABLE_STATS(
    ownname     => 'FINCORE',
    tabname     => 'LOAN_MASTER',
    estimate_percent => 100,
    method_opt   => 'FOR ALL COLUMNS SIZE AUTO FOR COLUMNS SIZE 254
                     LOAN_STATUS PRODUCT_TYPE RISK_GRADE',
    cascade      => TRUE,
    no_invalidate => FALSE
);
```

**Validation:** Re-run AWR-captured versions of R-01 and R-02 in a staging environment cloned from production, compare execution plans before/after statistics refresh.

---

### Phase 2 — Composite Index Deployment (Weeks 2–3)

New indexes built using `ONLINE` option (no DML blocking during build):

```sql
-- Index for regulatory report primary filter path
CREATE INDEX IDX_LM_STATUS_RPTDATE
    ON LOAN_MASTER(LOAN_STATUS, REPORT_DATE)
    TABLESPACE IDX_DATA
    ONLINE
    PARALLEL 4
    NOCOMPRESS;

-- Covering index for COLLATERAL_REGISTER join
-- Including COLLATERAL_VALUE avoids table access for this join
CREATE INDEX IDX_CR_LOAN_STAT
    ON COLLATERAL_REGISTER(LOAN_ID, COLLATERAL_STATUS)
    INCLUDE (COLLATERAL_VALUE)   -- (Oracle 11g+: index-organized covering)
    TABLESPACE IDX_DATA
    ONLINE;
```

**DML impact measurement:** After each index creation, INSERT/UPDATE throughput on `LOAN_MASTER` and `COLLATERAL_REGISTER` measured in staging over a 4-hour window with production-representative DML load. Acceptable threshold: < 5% DML throughput reduction.

Measured impact:
- `LOAN_MASTER` INSERT throughput: −2.1% (within threshold)
- `COLLATERAL_REGISTER` INSERT throughput: −3.8% (within threshold)

---

### Phase 3 — Query Rewriting (Weeks 2–4, parallel)

**R-03 Stored Procedure — cursor loop → set-based:**

Original (cursor loop):
```sql
-- BEFORE: cursor processing 2.4M loans individually
OPEN loan_cur FOR SELECT LOAN_ID, OUTSTANDING_BALANCE, ... FROM LOAN_MASTER WHERE ...;
LOOP
    FETCH loan_cur INTO v_loan_id, v_balance, ...;
    EXIT WHEN loan_cur%NOTFOUND;
    -- calculate accrual
    INSERT INTO ACCRUAL_LEDGER VALUES (v_loan_id, calculated_accrual, ...);
END LOOP;
CLOSE loan_cur;
```

Rewritten (set-based):
```sql
-- AFTER: single INSERT ... SELECT for entire portfolio
INSERT /*+ APPEND PARALLEL(al, 4) */ INTO ACCRUAL_LEDGER (LOAN_ID, ACCRUAL_AMOUNT, ...)
SELECT
    l.LOAN_ID,
    ROUND(l.OUTSTANDING_BALANCE * l.DAILY_RATE * :days_in_period, 2) AS ACCRUAL_AMOUNT,
    ...
FROM LOAN_MASTER l
WHERE l.LOAN_STATUS IN ('ACTIVE', 'RESTRUCTURED')
  AND l.CALC_DATE = :calc_date;

COMMIT;
```

The `APPEND` hint enables direct-path insert (bypasses buffer cache, writes directly to data file extent). This is appropriate here because: (a) the `ACCRUAL_LEDGER` partition being written to has no concurrent readers during the batch window, (b) full backup occurs before batch start.

**R-02 Inline View — WITH clause materialization:**

```sql
-- AFTER: WITH clause forces materialization of the common subquery
WITH loan_base AS (
    SELECT /*+ MATERIALIZE */
        l.LOAN_ID,
        l.OUTSTANDING_BALANCE,
        l.RISK_GRADE,
        l.PRODUCT_TYPE,
        l.BRANCH_CODE
    FROM LOAN_MASTER l
    WHERE l.LOAN_STATUS IN ('ACTIVE', 'DELINQUENT', 'RESTRUCTURED')
      AND l.REPORT_DATE = :reportDate
),
-- ... rest of query using loan_base twice without re-scanning LOAN_MASTER
```

The `/*+ MATERIALIZE */` hint prevents Oracle from merging the CTE back into the outer query — it is computed once and stored as a temp table segment for the duration of the query.

### Phase 4 — Nightly Batch Redesign (Weeks 3–5)

**Pagination pattern:** The Spring Batch `JpaPagingItemReader` with `ORDER BY LOAN_ID OFFSET n` was replaced with a **cursor-based pagination** approach using `LOAN_ID > :lastProcessedId ORDER BY LOAN_ID` (keyset pagination). This eliminates the progressive offset cost — every page scan starts at a known position via the primary key index.

```java
// BEFORE — offset pagination (gets slower with each page)
@Bean
public JpaPagingItemReader<Loan> loanReader() {
    return new JpaPagingItemReaderBuilder<Loan>()
        .name("loanReader")
        .entityManagerFactory(emf)
        .queryString("SELECT l FROM Loan l ORDER BY l.loanId")
        .pageSize(1000)
        .build();
}

// AFTER — keyset pagination with native SQL cursor
@Bean
public JdbcCursorItemReader<LoanRow> loanCursorReader(DataSource ds) {
    return new JdbcCursorItemReaderBuilder<LoanRow>()
        .name("loanCursorReader")
        .dataSource(ds)
        .sql("""
            SELECT LOAN_ID, OUTSTANDING_BALANCE, DAILY_RATE, LOAN_STATUS, PRODUCT_TYPE
            FROM LOAN_MASTER
            WHERE LOAN_STATUS IN ('ACTIVE', 'RESTRUCTURED')
              AND CALC_DATE = ?
            ORDER BY LOAN_ID
            """)
        .preparedStatementSetter(ps -> ps.setDate(1, calcDate))
        .rowMapper(loanRowMapper())
        .fetchSize(5000)      // Oracle prefetch hint
        .build();
}
```

The `JdbcCursorItemReader` opens a single database cursor and streams rows in order — one round trip for 2.4M rows versus 2,400 round trips with paging.

**Batch INSERT:** Spring Batch `JdbcBatchItemWriter` with batch size of 500:

```java
@Bean
public JdbcBatchItemWriter<AccrualEntry> accrualWriter(DataSource ds) {
    return new JdbcBatchItemWriterBuilder<AccrualEntry>()
        .dataSource(ds)
        .sql("INSERT INTO ACCRUAL_LEDGER (LOAN_ID, ACCRUAL_AMOUNT, CALC_DATE) VALUES (?, ?, ?)")
        .itemPreparedStatementSetter((entry, ps) -> {
            ps.setLong(1, entry.loanId());
            ps.setBigDecimal(2, entry.accrualAmount());
            ps.setDate(3, entry.calcDate());
        })
        .build();
}
```

`JdbcBatchItemWriter` sends batches of 500 INSERTs as a single `executeBatch()` call — 2.4M rows / 500 = 4,800 batch calls versus 2.4M individual calls.

### Phase 5 — Materialized Views (Weeks 5–6)

```sql
-- Pre-aggregated portfolio exposure — refreshed at 01:00 AM before report run
CREATE MATERIALIZED VIEW MV_PORTFOLIO_EXPOSURE_SUMMARY
    BUILD DEFERRED
    REFRESH COMPLETE ON DEMAND
    ENABLE QUERY REWRITE
AS
SELECT
    l.BRANCH_CODE,
    l.PRODUCT_TYPE,
    l.RISK_GRADE,
    l.REPORT_DATE,
    SUM(l.OUTSTANDING_BALANCE)                                          AS TOTAL_EXPOSURE,
    COUNT(l.LOAN_ID)                                                    AS LOAN_COUNT,
    SUM(CASE WHEN l.DAYS_PAST_DUE > 90 THEN l.OUTSTANDING_BALANCE
             ELSE 0 END)                                                AS NPL_BALANCE,
    AVG(l.INTEREST_RATE)                                                AS AVG_RATE
FROM LOAN_MASTER l
WHERE l.LOAN_STATUS IN ('ACTIVE', 'DELINQUENT', 'RESTRUCTURED')
GROUP BY l.BRANCH_CODE, l.PRODUCT_TYPE, l.RISK_GRADE, l.REPORT_DATE;
```

Refresh is triggered by the Spring Batch job scheduler 90 minutes before the report generation job begins:

```java
// Spring Batch step: refresh materialized views before report
@Bean
public Step refreshMaterializedViewsStep(JdbcTemplate jdbc) {
    return stepBuilderFactory.get("refreshMvStep")
        .tasklet((contribution, context) -> {
            jdbc.execute("BEGIN DBMS_MVIEW.REFRESH('MV_PORTFOLIO_EXPOSURE_SUMMARY', 'C'); END;");
            jdbc.execute("BEGIN DBMS_MVIEW.REFRESH('MV_PROVISIONING_BASE', 'C'); END;");
            return RepeatStatus.FINISHED;
        })
        .build();
}
```

## H. PostgreSQL Section — Aurora Optimization

Following the [AWS migration](./legacy-java-aws-migration), the same schema was running on Aurora PostgreSQL. Some Oracle-specific optimizations required re-implementation using PostgreSQL tooling.

### Diagnostic Tooling — PostgreSQL Equivalents

| Oracle | PostgreSQL Equivalent |
|--------|----------------------|
| AWR / ASH | `pg_stat_statements`, `pg_stat_activity` |
| DBMS_XPLAN | `EXPLAIN (ANALYZE, BUFFERS, FORMAT TEXT)` |
| V$SQL | `pg_stat_statements` |
| DBA_HIST_SQL_STAT | `pg_stat_statements` (requires `pg_stat_statements.track_planning = on`) |
| Oracle histograms | PostgreSQL MCV (Most Common Values) + histograms in `pg_stats` |

### Key PostgreSQL-Specific Findings

**Sequential scan on Aurora R-01 equivalent:**

```sql
EXPLAIN (ANALYZE, BUFFERS)
SELECT branch_code, product_type, risk_grade,
       SUM(outstanding_balance), COUNT(loan_id)
FROM loan_master
WHERE loan_status IN ('ACTIVE', 'DELINQUENT', 'RESTRUCTURED')
  AND report_date = '2024-09-30'
GROUP BY branch_code, product_type, risk_grade;
```

Output showed `Seq Scan on loan_master (cost=0.00..485,322.00 rows=2148320)` — PostgreSQL planner estimated `loan_status IN (...)` would return 100% of rows (no MCV statistics).

**Fix:**
```sql
-- Refresh statistics with extended histogram
ANALYZE loan_master;

-- Verify MCV statistics were captured
SELECT most_common_vals, most_common_freqs
FROM pg_stats
WHERE tablename = 'loan_master' AND attname = 'loan_status';
```

After `ANALYZE`, planner correctly estimated `loan_status IN (...)` at 78% selectivity and switched to an index scan on `(loan_status, report_date)`.

**PostgreSQL partial index — Aurora optimization:**

```sql
-- Partial index covering only active portfolio (78% of rows excluded)
CREATE INDEX CONCURRENTLY idx_lm_active_rptdate
    ON loan_master(report_date, branch_code, product_type)
    WHERE loan_status IN ('ACTIVE', 'DELINQUENT', 'RESTRUCTURED');
```

Index size: 38% of a full index on the same columns — reduced storage and maintenance overhead.

**Parallel query — Aurora PostgreSQL:**

```sql
-- Enable parallel query for the batch aggregation
SET max_parallel_workers_per_gather = 4;
SET parallel_tuple_cost = 0.01;

EXPLAIN (ANALYZE, BUFFERS)
SELECT loan_id, outstanding_balance * daily_rate * $1 AS accrual_amount
FROM loan_master
WHERE loan_status IN ('ACTIVE', 'RESTRUCTURED')
  AND calc_date = $2;
```

Parallel query reduced the full-table scan in the accrual batch from 380 seconds to 91 seconds on `db.r6g.xlarge` (4 vCPU worker threads).

**pg_stat_statements — top queries by total time:**

```sql
SELECT
    LEFT(query, 80)   AS query_preview,
    calls,
    ROUND(total_exec_time::numeric, 2) AS total_ms,
    ROUND(mean_exec_time::numeric, 2)  AS mean_ms,
    ROUND(stddev_exec_time::numeric, 2) AS stddev_ms,
    rows
FROM pg_stat_statements
ORDER BY total_exec_time DESC
LIMIT 20;
```

This replaced Oracle AWR's "Top SQL by Elapsed Time" view as the primary diagnostic starting point post-migration.

## H. Trade-offs & Engineering Decisions

### Complexity Introduced

**Materialized view refresh dependency:** The regulatory report now has an upstream dependency — the materialized view refresh job must complete before the report generation job starts. This was implemented as a Spring Batch job dependency chain. If the MV refresh fails (e.g., due to a locked table), the report job is blocked and an alert fires. This is an operational complexity that did not exist before.

**Set-based accrual procedure:** The cursor-loop-to-set-based transformation eliminated the ability to apply per-loan business logic exceptions that were previously coded inline in the cursor loop. Three exception cases (restructured loan special accrual, foreign currency adjustment, suspended accrual) had to be extracted into a separate post-processing step. This is architecturally cleaner but required careful analysis to ensure no exceptions were missed.

**Direct-path INSERT:** The `APPEND` hint on the accrual INSERT bypasses the undo tablespace. This means the INSERT cannot be rolled back after commit without full table restoration from backup. Mitigated by running a pre-batch checkpoint and keeping the batch in a transaction until the full set is validated.

### What Was Sacrificed

**Statistics maintenance overhead:** Extended histograms on `LOAN_MASTER.LOAN_STATUS` and `PRODUCT_TYPE` must be refreshed whenever data distribution changes significantly. This added a scheduled statistics job (nightly, after batch completes) that was not previously required. Failure to maintain statistics causes plan regression — this became a monitored operational task.

**Index maintenance overhead on DML:** Two new composite indexes on `LOAN_MASTER` add approximately 2.1% overhead to loan INSERT operations. At 18M daily DML operations, this is acceptable. At 5× growth, this would need re-evaluation.
